# CPU Offload:如何让"装不下"的大模型跑在单卡上

> 本文解释本项目解决单卡显存瓶颈的核心手段——CPU offload 的原理、效率、以及**什么场景适合、什么场景是无奈之举**。配套踩坑与配置见 [`patch_flux2_te.py`](../../poc/docker/patch_flux2_te.py) 与 README「显存管理」段。

---

## 1. 问题:双模型塞不进单卡

FLUX.2-dev = **32B transformer + Mistral-Small-24B 文本编码器**,两个独立模型。L40S 标称 48GB、PyTorch 进程实际可用仅 ~44.4GB。两个模型若同时驻留 GPU,峰值约 44GB,超出上限 → OOM。

---

## 2. 原理:模型权重只是"住在哪"的问题

一个神经网络 = 一堆权重张量(数字)。张量有个 `.device` 属性(`cpu` / `cuda`):

- `model.to('cpu')` = 把权重从**显存(VRAM)**拷到**内存(RAM)**,释放显存那份。
- `model.to('cuda')` = 反向拷回。
- **数字不变,只换物理位置**——像同一份文档在 U盘 或 硬盘间移动,内容一致。

约束:**计算必须在数据所在的设备上做**。GPU 只能算显存里的数据。所以"搬走"一个模型的前提是——**这段时间它确实用不到**。

这就是 offload 能来回的原因:offload 到 CPU 后,需要时再 upload 回 GPU,完全对称、可无限次。

---

## 3. 效率:时间换空间,瓶颈是 PCIe

搬运走 **PCIe 总线**(GPU↔CPU 的物理通道),这是成本来源:

| 通道 | 带宽 | 相对速度 |
|------|------|---------|
| GPU 内部显存 | ~800 GB/s+ | 基准(GPU 算数据的速度) |
| **PCIe 4.0 x16(搬运通道)** | **~25–32 GB/s** | 慢 20–30 倍 |
| CPU 内存 | ~50–100 GB/s | |

**单次搬运其实不慢**(亚秒~秒级):
- Mistral fp8(24GB)搬一次 ≈ 24 ÷ 28 ≈ **< 1s**
- transformer fp8(33GB)搬一次 ≈ **~1.2s**

**致命的不是单次搬运,是"在热循环里反复搬同一个模型"** —— 那样 GPU 大量时间在等 PCIe 而非计算,利用率崩。

### 划不划算 = 搬运频率 vs 搬完能算多久

| 场景 | 搬运频率 | 开销占比 | 结论 |
|------|---------|---------|------|
| **训练(本项目)** | Mistral 训练前搬一次,之后 1000+ 步零搬运 | <1s / 几十分钟 ≈ 0% | 净赚 |
| **推理(ComfyUI `--lowvram`)** | 每张图 TE/transformer/VAE 各搬 1–2 次 | 2–4s / 35s ≈ 10% | 可接受,换"三大模型不必同时驻显存" |

**原则:挑"接下来很长一段时间用不到"的模型搬走**,让一次搬运成本被后续大量计算摊薄。反例:每步都搬还要用的模型 = PCIe 拖垮 GPU。

> 本项目两个 patch 之所以高效,不是"搬得多",而是**"搬得准"**——在正确的时机(prepare 前)搬走一个正确的(一次性的)模型。见 [patch_flux2_te.py](../../poc/docker/patch_flux2_te.py)。

---

## 4. 本项目怎么用的(两个 patch)

1. **文本编码器用完即卸载(省 ~24GB,决定性)**
   扩散训练里 TE 是**一次性**的:把所有 caption 编码成 embedding 缓存后,训练循环再不需要它。
   踩坑:ai-toolkit 只在采样步卸载 TE,而 `prepare_accelerator`(搬 transformer 上 GPU)在采样之前 → 那一刻两模型同在 GPU = OOM。patch2 把卸载提前到 prepare 之前,腾出 24GB。

2. **CPU 上先量化再上 GPU(避开加载峰值)**
   错误顺序:先把 bf16 全模型(48GB)搬 GPU,再在 GPU 量化 → 搬上去那一瞬就爆。
   正确顺序:CPU 内存里先量化(48→24GB),再把瘦身版搬上 GPU(CPU 有 128GB,扛得住)。patch1 调换了这个顺序。

结果:训练显存从"爆 44GB"稳定到 **36GB / 44GB 上限**,余量 8GB。

---

## 5. 适用性:看模型有没有"长时间闲置的部分"

真正决定 offload 好不好用的,**不是"多模态 vs LLM",而是模型结构里有没有"用完就长时间闲置"的组件**:

| 模型类型 | 有无闲置部分 | offload 定位 |
|---------|------------|-------------|
| **多模态扩散(本项目)** | ✅ 有(TE→transformer→VAE 接力,时间不重叠) | **好场景**,近乎免费午餐,能省就省 |
| **稠密 LLM** | ❌ 无(单一 transformer 逐 token,每层每步都用) | **无奈之举**,硬 offload 层权重必落热循环 → 慢 5–20 倍 |
| **MoE LLM** | ✅ 有(每步只激活部分专家,未激活专家可卸) | 好场景,业界做专家 offload |

**为什么 LLM 更难**:
- 没有天然闲置的子模型,offload 的层权重每 token 都要访问 → 每步触发 PCIe 搬运。
- LLM 的大头常是 **KV cache 而非权重**,offload KV cache 更糟(每步增长 + 每步读)。
- 所以 LLM 界更偏好**不搬运**的方案:量化(压小权重)、张量并行(拆到多卡)、PagedAttention(省 KV cache)。offload 是"实在装不下才用,且明知会慢"。

> **可迁移的核心判断**:"识别出哪个组件是一次性/长时间闲置的、可安全搬走",比"我会用 offload"值钱得多。这是本项目在 FLUX.2 上的经验能推广到其他大模型场景的关键。
